# Repository Map — nikki-san

This document describes the **directory structure of the repository**.  
Its purpose is to help AI assistants understand **where files are located** and avoid incorrect assumptions.

This project uses **Hugo with the Stack theme**.

---

# Repository Root

Important directories:
- `content/`
- `layouts/`
- `assets/`
- `static/`
- `data/`
- `docs/`

A local `themes/` directory may be absent because the project uses **Hugo Modules**.

---

# content/

The `content/` directory contains all Hugo content.

Main structure:

```text
content/
├─ posts/
└─ archives/
```

## `content/posts/`

This directory contains all blog posts.  
Each post uses a **bundle structure**.

Structure:

```text
content/posts/<POST_ID>/index.md
```

Example:

```text
content/posts/6256/index.md
```

`POST_ID` corresponds to the original FC2 blog entry ID.

Assets for a post may also be stored inside the same directory.

Example:

```text
content/posts/6256/
├─ index.md
└─ image.jpg
```

## `content/archives/`

Archive pages are generated automatically.

Structure:

```text
content/archives/YYYY/_index.md
content/archives/YYYY/MM/_index.md
```

These files are generated by `rebuild_archives.py` and should not normally be edited manually.

---

# layouts/

The `layouts/` directory contains **theme overrides** and project-specific templates.

If a file exists here, it overrides the corresponding behavior from the Stack theme.

Common locations:
- `layouts/_default/`
- `layouts/section/`
- `layouts/_partials/`
- `layouts/partials/`

Important note:
- With the current Stack upstream structure, underscore-prefixed partial paths may matter.
- Do not assume that all partial overrides belong under `layouts/partials/`.

Examples relevant to this project:
- page templates such as `layouts/index.html`
- single-page templates such as `layouts/_default/single.html`
- archive templates such as `layouts/section/archives.html`
- recent comments template such as `layouts/_default/recent-comments.html`
- footer override paths under `layouts/_partials/footer/` if the project follows the upstream Stack structure

---

# assets/

The `assets/` directory contains files processed by Hugo Pipes.

Typical contents:
- `assets/scss/`

SCSS files may be compiled into CSS during the build process.

Example:
- `assets/scss/custom.scss`

---

# static/

The `static/` directory contains files copied directly to the final site.

Examples:
- `static/css/`
- `static/js/`
- `static/images/`

Files here appear in the final site without processing.

Example:
- `static/css/prism.css`
- `static/js/prism.js`

---

# data/

The `data/` directory contains structured data used by Hugo templates.

Example:
- `data/archives/months.json`

This file is generated by `rebuild_archives.py`.

Templates may load this data using Hugo's `.Site.Data`.

Other generated data files may also exist for features such as comments or recent comments.

---

# docs/

The `docs/` directory contains documentation used primarily by AI assistants.

Files include:
- `AI_CONTEXT.md`
- `ARCHITECTURE.md`
- `AI_DEBUG_GUIDE.md`
- `SCRIPTS.md`
- `REPO_MAP.md`
- `KNOWN_ISSUES.md`
- `AI_TASK_HISTORY.md`

These documents describe:
- system overview
- architecture
- debugging rules
- scripts
- repository layout
- notable historical decisions

---

# Scripts

Python scripts may exist in the repository root.

Examples:
- `fc2_export_to_hugo.py`
- `extract_comments.py`
- `rebuild_archives.py`

These scripts are not part of Hugo but generate content used by the site.

---

# Important Rules for AI

When referencing files:
1. confirm the directory structure
2. prefer existing overrides in `layouts/`
3. do not assume files exist unless confirmed
4. do not assume `themes/` exists in a Hugo Modules project

When debugging rendering issues, check in this order:
1. `layouts/` overrides
2. content files
3. data files
4. assets or static files
5. upstream theme behavior

---

# Notes

The repository is designed to keep:
- post URLs stable
- build output deterministic
- manual maintenance simple

AI suggestions should respect these constraints.
